# Lesson 68: The File Edit Model — old_string → new_string

## The Core Design Decision

How should an AI agent edit files? There are several approaches:

1. **Line-based**: "Replace lines 15-20 with this content"
2. **Diff/patch**: "Apply this unified diff"
3. **Full rewrite**: "Here's the entire new file"
4. **String replacement**: "Find this exact string, replace with this"

Claude Code chose option 4 — **string replacement** — and it's the single most important design decision in the file editing system. This lesson explains why, how it works, and what tradeoffs it involves.

---

## Why Not Line Numbers?

Line-based editing seems natural: "Replace line 15 with `const x = 2;`." But it breaks catastrophically in practice:

```
Turn 1: Model reads file (100 lines)
Turn 2: Model edits line 50 (inserts 3 lines → file now 103 lines)
Turn 3: Model wants to edit "line 80" — but line 80 shifted to line 83
```

The model's mental map of line numbers becomes stale after every edit. Worse, when multiple edits happen in a single turn, each one shifts the line numbers for all subsequent edits.

String replacement doesn't have this problem. The target string `"function oldName("` is the same regardless of what lines were added above it.

---

## Why Not Unified Diffs?

Diffs are powerful but fragile when generated by language models:

```diff
--- a/src/utils.ts
+++ b/src/utils.ts
@@ -14,7 +14,7 @@
   const config = loadConfig();
-  const result = process(config);
+  const result = processAsync(config);
   return result;
```

Problems:
- Models frequently get line numbers in the `@@` header wrong
- Context lines must exactly match the file (models hallucinate whitespace)
- Multi-hunk diffs are error-prone to generate
- The model must generate characters that don't change (wasteful tokens)

String replacement avoids all of these. The model only specifies what changes.

## The Zod Input Schema

```typescript
const inputSchema = z.strictObject({
  file_path: z
    .string()
    .describe("The absolute path to the file to edit"),
  old_string: z
    .string()
    .describe("The text to search for — must match exactly"),
  new_string: z
    .string()
    .describe("The text to replace old_string with"),
  replace_all: z
    .boolean()
    .optional()
    .default(false)
    .describe(
      "If true, replaces ALL occurrences of old_string. " +
      "Default false (replaces only the first unique match)."
    ),
});
```

Four fields. That's it. The simplicity is intentional — fewer fields means fewer ways to get it wrong.

## How It Works: Step by Step

### Basic Replacement

```typescript
function applyEdit(
  fileContent: string,
  oldString: string,
  newString: string,
  replaceAll: boolean
): string {
  if (replaceAll) {
    return fileContent.split(oldString).join(newString);
  }

  const index = fileContent.indexOf(oldString);
  if (index === -1) {
    throw new ToolError(
      `old_string not found in file. Make sure it matches exactly, ` +
      `including whitespace and indentation.`
    );
  }

  // Check for uniqueness (single-match mode)
  const secondIndex = fileContent.indexOf(oldString, index + 1);
  if (secondIndex !== -1) {
    throw new ToolError(
      `old_string appears multiple times in file. ` +
      `Include more surrounding context to make it unique, ` +
      `or use replace_all: true.`
    );
  }

  return (
    fileContent.substring(0, index) +
    newString +
    fileContent.substring(index + oldString.length)
  );
}
```

The algorithm is deliberately simple:
1. Find the `old_string` in the file content
2. Verify it appears exactly once (unless `replace_all` is true)
3. Replace it with `new_string`

### Uniqueness Requirement

The uniqueness check is critical. Without it, the model might intend to edit one `return null;` but accidentally change a different one. By requiring unique matches, the system forces the model to include enough context:

```typescript
// Too ambiguous — appears on many lines:
{ old_string: "return null;", new_string: "return undefined;" }

// Unique — includes surrounding context:
{
  old_string: "  if (!user) {\n    return null;\n  }",
  new_string: "  if (!user) {\n    return undefined;\n  }"
}
```

The error message teaches the model: *"Include more surrounding context to make it unique."*

## Example Walkthrough

Let's trace through a complete edit. Starting file (`app.ts`):

```typescript
import express from "express";

const app = express();

app.get("/health", (req, res) => {
  res.json({ status: "ok" });
});

app.get("/users", (req, res) => {
  const users = getUsers();
  res.json(users);
});

app.listen(3000);
```

The model wants to add error handling to the `/users` endpoint:

```json
{
  "file_path": "/app.ts",
  "old_string": "app.get(\"/users\", (req, res) => {\n  const users = getUsers();\n  res.json(users);\n});",
  "new_string": "app.get(\"/users\", async (req, res) => {\n  try {\n    const users = await getUsers();\n    res.json(users);\n  } catch (err) {\n    res.status(500).json({ error: \"Failed to fetch users\" });\n  }\n});"
}
```

Execution:
1. Read `app.ts` content
2. Search for `old_string` — found at position 108
3. Check for second occurrence — none found, it's unique
4. Replace: content before (0-108) + `new_string` + content after
5. Write result back to file

Result:

```typescript
import express from "express";

const app = express();

app.get("/health", (req, res) => {
  res.json({ status: "ok" });
});

app.get("/users", async (req, res) => {
  try {
    const users = await getUsers();
    res.json(users);
  } catch (err) {
    res.status(500).json({ error: "Failed to fetch users" });
  }
});

app.listen(3000);
```

The surrounding code is untouched. Only the matched region changed.

## The replace_all Flag

Sometimes you genuinely need to change every occurrence:

```json
{
  "file_path": "/src/config.ts",
  "old_string": "http://localhost:3000",
  "new_string": "https://api.example.com",
  "replace_all": true
}
```

With `replace_all: true`, the uniqueness check is skipped and every occurrence is replaced. This is useful for:

- Renaming variables or functions
- Updating URLs or configuration values
- Changing import paths

```typescript
// Replace all occurrences of a function name
{
  old_string: "getUserById",
  new_string: "findUserById",
  replace_all: true
}
```

## Edge Cases the Model Must Handle

### Empty old_string (Insertion)

If `old_string` is empty, what happens? The tool rejects it:

```typescript
if (oldString.length === 0) {
  throw new ToolError(
    "old_string cannot be empty. To insert at a specific location, " +
    "include the surrounding text in old_string and new_string."
  );
}
```

To insert text, the model must include context around the insertion point:

```json
{
  "old_string": "import express from \"express\";\n\nconst app",
  "new_string": "import express from \"express\";\nimport cors from \"cors\";\n\nconst app"
}
```

### Identical old_string and new_string

```typescript
if (oldString === newString) {
  throw new ToolError(
    "old_string and new_string are identical. No changes needed."
  );
}
```

This catches a no-op that would waste a tool call.

### Whitespace Sensitivity

The match is whitespace-sensitive. This fails:

```json
{
  "old_string": "function hello() {\n  return 'world';\n}",
  "new_string": "..."
}
```

...if the file actually uses tabs instead of spaces. Lesson 69 covers how quote normalization helps with some of these issues, but whitespace sensitivity is fundamentally correct — it prevents ambiguous matches.

---

## Why This Model Works So Well

The `old_string → new_string` model succeeds because it aligns with how language models think:

1. **Context as anchor**: The model naturally includes surrounding context to identify the right location
2. **No counting**: No line numbers, column numbers, or offsets to get wrong
3. **Idempotent verification**: The uniqueness check catches ambiguity before damage is done
4. **Minimal output**: The model only generates the changed region, saving tokens
5. **Human readable**: A reviewer can immediately see what changed by comparing `old_string` and `new_string`

---

## Key Takeaways

1. **String replacement beats line-based editing** because it's robust to line number changes from prior edits.

2. **Uniqueness is enforced** — the `old_string` must appear exactly once (unless `replace_all` is true), forcing the model to include enough context.

3. **Error messages teach** — when a match fails, the message tells the model *how* to fix it.

4. **The model only generates what changes**, reducing token waste and hallucination risk.

5. **replace_all** enables batch operations like renames and URL updates.
